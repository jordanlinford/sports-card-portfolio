1) Slow down HARD + remove concurrency
Global limits (non-negotiable):
	•	Max 1 scrape worker at a time (globally)
	•	Max 1 page request every 5–10 seconds
	•	Max 2 pages per query, not 3+
	•	Add random jitter (+/- 2–3 seconds)

This alone fixes a lot of blocking.

2) Cache everything (and don’t refetch if you don’t have to)
	•	Cache key = queryHash
	•	TTL = 24 hours (or longer)
	•	If you already have 15+ comps, don’t refresh often
	•	If you have <5 comps, refresh later, not immediately

The #1 cause of blocks is refetching the same query repeatedly.

3) Reuse sessions (cookies) and send real browser headers
If you’re doing raw fetches:
	•	Persist cookies per “scrape session”
	•	Use consistent headers:
	•	Accept-Language
	•	Accept
	•	Referer
	•	User-Agent (don’t rotate every request)

Rotating UA too aggressively can look more botty, not less.

4) Detect blocks explicitly and mark them correctly
Right now you’re showing “HIGH” confidence while also saying fallback comps + 2 sold. That’s a logic bug.

If eBay blocked:
	•	comps.status = "blocked"
	•	confidence = "LOW"
	•	soldCount should reflect only what you actually used

Also store:
	•	blockReason: "429" | "captcha" | "robot-check" | "unexpected-html"

5) Don’t scrape in the user request
You already planned this — keep it.
	•	Return partial quickly
	•	Queue background job
	•	Poll status

6) Consider a browser-based fetch only when needed (fallback tier)
If HTML fetch is blocked:
	•	Try a Playwright “real browser” job only for high-value queries or Pro users
	•	Strictly rate limit it

This is heavier, but it’s the only way to beat some of eBay’s protections without changing sources.

Note: this can impact hosting costs and deploy size; keep it optional and behind limits.
